# src/core/weight/

Weight entry domain: pure types and transformations for a single kitten's weight within a feeding session.

## Outputs / Contract

- `WeightEntry` interface — `{ id, sessionId, kittenId, grams, timestamp, clientWriteId }`
- `NullWeightEntry` — Null Object substitutable wherever `WeightEntry` is expected
- `weightEntryId(sessionId, kittenId) → string` — deterministic composed id; same inputs always produce the same id
- `createWeightEntry({ sessionId, kittenId, grams, timestamp, clientWriteId }) → WeightEntry` — produces a fresh entry with composed id
- `validateGrams(grams) → { valid, errors[] }` — pure validation

## Dependencies

None at runtime.

## Invariants

- `id === \`${sessionId}:${kittenId}\`` — there is exactly one canonical entry per (session, kitten) pair; updates are upserts via the same id
- `grams` is a positive integer; decimal grams and zero/negative are invalid
- `grams` upper bound is 5000 (sanity cap; typo guard against extra digits)
- `timestamp` is milliseconds since epoch (the moment of entry, not the session start)
- `clientWriteId` is a fresh UUID per write; generated by the shell at write time
- `NullWeightEntry.id === ''` — consumers detect "no entry for this kitten yet" by checking id, not by checking grams (a real entry of 0 grams is invalid anyway, but rule applies regardless)

## Why composed id?

Each kitten in a session has at most one weight entry. Composed id makes upsert trivial (`db.weight_entries.put(entry)`) and removes the need for an "is there one already?" lookup. Sync (Phase 4) can still use `clientWriteId` for idempotency on the wire, independent of the local primary key.
